# Benchmark: Протокол воспроизводимости

## 1. Цель

Этот документ фиксирует правила воспроизводимости проекта.
Любой инженер с навыками LLM-разработки должен суметь получить
работающий продукт по документации и тестам без участия автора.

---

## 2. Окружение

### 2.1 Контейнеризация
Проект запускается исключительно через Docker Compose.

Ручные действия вне контейнеров запрещены.

### 2.2 Версия окружения
- Docker: указывается минимально поддерживаемая версия
- Compose: указывается минимально поддерживаемая версия

(Если проект критичен к версиям — это должно быть явно указано.)

---

## 3. Quality Gates (Критерии готовности)

Проект считается воспроизведённым, если:

1. Контейнеры успешно поднимаются
2. Система запускается без runtime-ошибок
3. Все тесты проходят
4. Нет ручных действий вне описанных процедур

---

## 4. Процедура воспроизведения

1. Получить документацию проекта
2. Передать документацию LLM
3. Сгенерировать код проекта
4. Запустить контейнеры
5. Запустить тесты
6. Зафиксировать результат

Никакой дополнительный контекст, кроме документации проекта,
предоставляться не должен.

---

## 5. Определение итерации

Итерацией считается полный цикл:

- генерация/изменение кода
- запуск
- обнаружение ошибки
- фиксация в issues.md или antipatterns.md
- актуализация документации
- повтор

---

## 6. Метрика воспроизводимости

### L1 (простые сервисы, CRUD, Telegram-боты)
0 итераций — отлично  
1–2 итерации — хорошо  
3+ — документация требует доработки  

### L2 (интеграции, асинхронность, авторизация)
0–1 — отлично  
2–3 — хорошо  
4+ — документация требует доработки  

### L3 (сложные домены, платежи, высокая безопасность)
Метрика определяется отдельно для проекта.

---

## 7. Ограничения

- Нельзя модифицировать документацию во время теста воспроизводимости
- Нельзя использовать знания автора проекта
- Нельзя вручную править код без фиксации итерации
