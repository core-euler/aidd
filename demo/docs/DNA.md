# DNA проекта GameTODO Bot

## Цель
Создать Telegram-бота для управления задачами с игровой механикой: выполнение задач приносит опыт, просрочка наносит урон. При нулевом здоровье прогресс сбрасывается. Бот работает через inline-кнопки и хранит данные в PostgreSQL. Вся разработка ведётся по методике LLM-Driven, где спецификация — главный артефакт.

## Границы проекта
Входит:
- создание и управление задачами с дедлайнами
- игровая логика уровней, опыта и здоровья
- уведомления о дедлайне и просрочке
- статистика и статус персонажа
- хранение данных и логирование действий
- запуск через Docker Compose

Не входит:
- веб-интерфейс или мобильное приложение
- интеграция с внешними сервисами продуктивности
- платёжные функции и монетизация
- многопользовательские игровые механики

## Ключевые принципы
- Документация без кода и псевдокода
- Одна актуальная версия документации
- Модульность и изоляция компонентов
- Только Docker Compose для запуска
- Нельзя изменять работающий код без явной необходимости

## Основные артефакты документации
- SPEC.md — исходное техническое задание
- ARCHITECTURE.md — архитектура и границы модулей
- DATA_MODEL.md — сущности, поля и ограничения
- FLOWS.md — сценарии, состояния, переходы
- UI_TEXTS.md — тексты и состав экранов
- SCHEDULER.md — фоновые проверки и уведомления
- INFRA.md — окружение и эксплуатационные требования
- TEST_PLAN.md — перечень тестов в терминах поведения
- TEST_CONTRACT.md — контракт тестирования и критерии проверки
- ISSUES.md — известные проблемы и решения
- DNA_MAP.md — карта связей требований и тестов
- phases/ — этапы реализации
- ANTIPATTERNS.md — типичные ошибки и как их избегать

## Внешние интеграции
- Telegram Bot API: получение входящих сообщений и отправка ответов пользователю
- PostgreSQL: долговременное хранение пользователей и задач

## Риски и чувствительные зоны
- Обработка дедлайнов и часовых поясов
- Многократные просрочки и суммарный урон
- Одновременные действия пользователя и фоновых проверок
- Удержание актуальности данных при перезапуске

## Критерии готовности
- Все требования SPEC.md покрыты документацией
- Архитектура и модули имеют чёткие границы
- Сценарии, состояния и граничные случаи описаны словами
- План тестирования покрывает функциональные и edge cases
