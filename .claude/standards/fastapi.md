- Routers: request validation (via Pydantic), call service, return response

- Dependencies: injected via FastAPI `Depends()`, not global imports

## Configuration

- All settings via `pydantic_settings.BaseSettings`, source: `.env`
- Separate settings classes if needed: `DatabaseSettings`, `RedisSettings`, `AuthSettings`
- No defaults for secrets â€” app must fail to start if missing

## Schemas (Pydantic)

- Separate schemas for request and response: `UserCreate` (input) vs `UserResponse` (output)
- Use `model_config = ConfigDict(from_attributes=True)` for ORM compatibility
- Never expose internal fields (passwords, internal IDs) in response schemas
- Shared base schemas to avoid duplication: `UserBase` â†’ `UserCreate` / `UserResponse`

## Database

- SQLAlchemy 2.x async with `AsyncSession`
- Session provided via `Depends(get_db_session)` â€” yields session, auto-closes
- One session per request, no manual commit in routers (commit in service/repository)
- Alembic for migrations, auto-generated from models
- Connection pool: `pool_size=5`, `max_overflow=10` (tuned per load)

## Authentication

- JWT tokens via `python-jose` or `PyJWT`
- `Depends(get_current_user)` â€” extracts and validates token, returns user object
- Guard pattern: dependency raises `HTTPException(401)` if invalid
- Token in `Authorization: Bearer <token>` header
- Refresh token flow if session longevity needed

## Dependency Injection

- DB session: `Depends(get_db_session)`
- Current user: `Depends(get_current_user)`
- Services: `Depends(get_user_service)` â€” factory that receives session and builds service with repository
- Chain: router gets service via Depends â†’ service gets repository â†’ repository gets session
- Never instantiate services or repositories manually in routers

## Error Handling

- Custom exception classes inheriting from a base `AppError`
- Global exception handler registered via `app.add_exception_handler()`
- HTTP errors: use `HTTPException` with appropriate status codes
- Validation errors: FastAPI handles automatically via Pydantic (422)
- All errors logged via structlog with request context

## API Design

- RESTful naming: nouns for resources, HTTP verbs for actions
- Versioned routes: `/api/v1/...`
- Consistent response envelope or direct model returns (pick one, stay consistent)
- Pagination: `limit` + `offset` parameters, return `total` count
- Filtering/sorting via query parameters

## Health & Observability

- `GET /health` â€” checks DB and Redis connectivity, returns status or 503
- All requests logged: method, path, status code, duration_ms, user_id (if authenticated)
- structlog in JSON format, same as bot standards
- Request ID middleware: generate UUID per request, include in all log entries

## CORS

- Configured in middleware, not per-route
- Allowed origins from environment variable (comma-separated list)
- Credentials: enabled if auth cookies used
- Restrictive in production, permissive in development

## OpenAPI / Swagger

- Auto-generated by FastAPI (enabled by default)
- All routes have `summary` and `response_model`
- Group routes with `tags`
- Auth shown in Swagger via `HTTPBearer` security scheme

## Required

- Pydantic schemas for all request/response bodies
- Dependency injection for DB sessions, auth, and services
- Health check endpoint
- Alembic migrations (no raw SQL for schema changes)
- structlog for all logging
- CORS middleware configured from env
- Request validation via Pydantic (never manual parsing)
- Async everywhere â€” no blocking calls

## Forbidden

- Business logic in routers
- Raw SQL without SQLAlchemy (except rare performance-critical queries)
- Manual session management in routers
- Exposing internal model fields in API responses
- Hardcoded secrets or CORS origins
- Sync database drivers in async app
- Catching broad `Exception` without re-raising or logging
- Disabled OpenAPI in production (keep it, protect with auth if needed)
