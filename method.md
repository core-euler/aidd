# Методичка по LLM-Driven разработке (AIDD v2)

## Введение

Эта методичка описывает подход к разработке программного обеспечения с использованием LLM (Large Language Models), где главным артефактом является не код, а спецификация. Код генерируется нейросетью на основе документации.

**Главный принцип:** контекст — это наше главное оружие. Больше сути, меньше воды. Никакого кода в документации.

**Что изменилось во второй версии:**

Методичка дополнена практиками, которые стали стандартом индустрии: sub-agents и параллельная работа, MCP (Model Context Protocol), spec-driven development как именованная методология, context engineering как отдельная дисциплина, расширенные verification loops с акцентом на слепые зоны автогенерации тестов. Фундаментальные принципы первой версии не изменились — они проверены практикой и остаются ядром подхода.

---

## Роль человека в LLM-разработке

Человек — это тот, кто управляет системой. Он обязан понимать:

- **Архитектуру** — как компоненты связаны между собой
- **Инфраструктуру** — как всё развёрнуто и работает
- **Бизнес-логику** — что система делает и зачем

Человеку НЕ нужно знать:

- Фундаментальные алгоритмы (сортировки, деревья, графы)
- Синтаксис языка программирования
- Как устроены стрелочные функции или декораторы
- Тонкости реализации

**Архитектурное мышление важнее навыков кодинга.** Человек принимает решения на уровне "что делаем и как это устроено", нейросеть решает "как это написать в коде".

Это обязательное условие для работы по данной методологии.

**Главное ограничение методологии:**

AIDD не работает без квалифицированного человека-оркестратора. Не разработчик не сможет сделать качественный продукт по этой методике, даже имея лучшие LLM. Причина в том, что LLM прекрасно генерирует код, но не умеет:

- Принимать архитектурные решения с учётом долгосрочных последствий
- Определять, где модульность критична, а где избыточна
- Замечать, что сгенерированные тесты покрывают не то, что нужно
- Видеть, когда "работает на тестах" расходится с "работает в реальности"
- Контролировать границы изменений между параллельными агентами

Это не ограничение конкретной модели — это структурное свойство задачи. LLM работает в рамках поставленного контекста. Решение, какой контекст подать и как его поделить — на человеке. Эта мысль будет повторяться в каждом ключевом разделе методички, потому что она объясняет, почему AIDD — это не "AI пишет код за вас", а "вы управляете AI как инструментом".

## Часть 1. Фундаментальные правила

### 1.1. Никогда не писать код в документации

Это правило не имеет исключений (кроме одного — антипаттерны).

**Запрещено:**

- Код на любом языке программирования
- Псевдокод (даже он воспринимается нейросетью как директива)
- SQL-запросы
- Регулярные выражения
- Любые конструкции, похожие на код

**Разрешено:**

- Описание человеческим языком
- Блок-схемы и диаграммы
- Таблицы решений
- State-машины (визуально)

**Почему псевдокод тоже запрещён:**

Даже псевдокод воспринимается нейросетью как конкретная инструкция к действию. Она начинает буквально следовать этой структуре, вместо того чтобы самостоятельно найти оптимальное решение. Это опасно — получаем слепое следование вместо осмысленной генерации.

**Почему это критично:**

**Причина 1: Несогласованность**

Когда нейросеть видит код как инструкцию, она выполняет его буквально. Но этот код написан без знания полного контекста кодовой базы, без понимания того, как всё будет взаимодействовать. В результате получаются слепые кусочки лего, которые заранее не согласованы друг с другом. Это гарантированный источник ошибок.

**Причина 2: Засорение контекста**

Контекстное окно LLM ограничено. Каждый символ кода в документации — это место, которое могло бы занять полезная информация. Код многословен, спецификация компактна. Выбор очевиден.

**Единственное исключение:**

Антипаттерны в разделе issues — примеры того, как делать НЕ надо. Это помогает нейросети избежать типичных ошибок.

**Как описывать логику:**

Словами. Описываем что должно происходить, какие данные на входе, какие на выходе, какие граничные условия, какие ошибки возможны. Нейросеть сама решит, как это реализовать в коде.

Пример правильного описания:
> "Функция получает список пользователей. Отфильтровывает только активных. Сортирует по дате регистрации от новых к старым. Возвращает первые 10."

Пример неправильного (псевдокод):
> "for user in users: if user.active: result.append(user); return sorted(result)[:10]"

### 1.2. Всегда только одна актуальная версия документации

Не плодить файлы: spec-v1.md, spec-v2.md, spec-v3.md — это путь к хаосу.

**Почему:**

Спецификация никогда не идеальна изначально. Всегда будут неучтённые моменты, несовместимости, ошибки. Это нормально. Но создавать новую версию документации при каждом изменении — неверно.

**Как правильно:**

- Актуализировать оригинальный файл
- Вести отдельный changelog (лог изменений) — что изменилось и почему
- Предыдущие версии не сохранять
- Если изменение затрагивает один этап — актуализировать документацию этапа
- Если изменение затрагивает весь проект — актуализировать всё

История версий живёт в Git. Документ всегда отражает текущее состояние истины. Если нужно посмотреть, как было раньше — это работа для `git log`, а не для копий файлов в корне проекта.

### 1.3. Модульность и микросервисы

Стремимся к модульной архитектуре по одной причине: модули легче контролируются нейросетями.

Маленький изолированный сервис с чёткими границами — это то, что LLM может полностью держать в контексте и генерировать без ошибок.

**Дополнительная причина для AIDD v2:** модульность напрямую определяет, насколько эффективно работают параллельные sub-agents. Если модули плохо изолированы — субагенты пересекаются, ломают работу друг друга, и параллельность превращается в источник конфликтов. Подробнее в разделе про sub-agents.

---

### 1.4. Только Docker Compose

Никаких ручных запусков. Только контейнеризация.

- Docker Compose для оркестрации
- Отдельный Dockerfile для каждого сервиса
- Воспроизводимость окружения гарантирована

Это требование напрямую связано с принципом воспроизводимости: продукт должен собираться в чистом окружении из документации без участия автора. Без контейнеризации это невозможно проверить честно.

---

### 1.5. Не трогать работающий код

При добавлении новой функции явно указывать нейросети:

> "Ты реализуешь новую функцию. Ты наращиваешь её на существующую. Существующий код не меняешь — там всё работает."

Без этого указания нейросеть почти всегда начинает "улучшать" и "адаптировать" старый код. Это недопустимо.

Исключение: когда изменение старого кода явно требуется по задаче.

**Замечание про современные модели:** более мощные модели (Opus, аналогичные) лучше держат это правило, но не идеально. Даже им нужно явное указание границ. Слабые модели игнорируют это требование почти всегда — поэтому при работе со слабой моделью граница "не трогать" должна повторяться буквально в каждом промпте.

---

### 1.6. Разделение чатов по ролям и sub-agents

Каждый рабочий контекст с LLM — это отдельная роль с отдельной ответственностью.

**Принцип:** один контекст = одна зона ответственности.

**Типичное разделение:**

| Роль | Ответственность | Не занимается |
|------|-----------------|---------------|
| Архитектор | Планирование, структура, компоненты | Кодом, API, багами |
| API-специалист | Внешние интеграции, endpoints, документация API | Архитектурой, внутренней логикой |
| Разработчик | Реализация модулей по спецификации | Архитектурными решениями |
| Фиксер | Исправление конкретных багов | Новым функционалом |
| Интегратор | Связывание модулей между собой | Внутренней логикой модулей |

**Почему это важно:**

1. **Чистый контекст** — каждый рабочий контекст содержит только релевантную информацию
2. **Фокус** — нейросеть не пытается делать всё сразу
3. **Отслеживаемость** — понятно, где что обсуждалось
4. **Качество** — специализированный промпт даёт лучший результат

**Пример для API-интеграции:**

В контексте "API-специалист" создаётся отдельный файл документации: какие endpoints внешнего сервиса используются, как данные преобразуются в нашем проекте, куда они дальше идут. Этот файл потом используется контекстом "Разработчик" при реализации.

#### Sub-agents: эволюция разделения ролей

Современные среды разработки на базе LLM (Claude Code, аналоги) формализуют разделение ролей через механизм sub-agents. Это автоматический запуск отдельных агентов с собственным контекстом, изолированных друг от друга.

**Что даёт формализация:**

- Конфигурация агентов хранится в репозитории (например, `.claude/agents/`)
- Параллельный запуск нескольких агентов на разные модули
- Изоляция контекстов на уровне инфраструктуры, а не дисциплины
- Воспроизводимость состава ролей при смене разработчика

**Главный принцип параллельной работы:**

Изоляция по модулям важнее, чем изоляция по ролям. Когда параллельно работают несколько sub-agents, критично не то, какую роль они играют, а **на каких модулях работают**. Если два агента одновременно меняют код в пересекающихся модулях — будут конфликты, неконсистентность, сломанные интеграции.

**Из практики:**

Я использую несколько одновременных рабочих окон с sub-agents. Главное правило — модули, над которыми они работают, не должны пересекаться. Большие LLM сами по себе неплохо контролируют границы, но человек контролирует это намного лучше. Разделение задач между агентами — это архитектурное решение, которое нельзя делегировать модели целиком.

**Практический шаблон распределения:**

- Агент A → модуль авторизации
- Агент B → модуль платежей
- Агент C → модуль уведомлений

Все трое могут работать параллельно. Но если агенту A потребуется что-то от модуля платежей — он либо ждёт, либо работает только с интерфейсом (не с реализацией).

**Что нельзя делегировать:**

- Решение о границах модулей
- Решение, какие задачи действительно параллелятся
- Контроль того, что агенты не лезут на чужую территорию
- Финальную сверку результатов между агентами

Параллельность — мощный инструмент, но он умножает не только скорость, но и масштаб ошибок. Без человека-оркестратора три агента создадут проблем втрое больше, чем один.

### 1.7. Context Engineering vs Prompt Engineering

Два разных уровня работы с LLM, которые часто смешивают.

**Prompt engineering** — формулировка отдельного запроса. Как сформулировать инструкцию, чтобы получить нужный результат на одном промпте. Это уровень предложения.

**Context engineering** — управление всем контекстным окном на протяжении сессии или всего проекта. Что грузим, что выгружаем, как сжимаем длинные документы, когда чистим контекст, как разделяем информацию между агентами. Это уровень архитектуры взаимодействия.

**Эта методичка — про context engineering.** Все правила (одна актуальная версия документации, разделение на этапы, карта модулей, разделение ролей) — это приёмы управления контекстом, а не приёмы формулировки промптов.

**Базовые приёмы context engineering:**

1. **Селективная загрузка.** При работе над модулем загружаем только связанные модули (по карте модулей), а не всё подряд.

2. **Иерархия документов.** spec.md (всегда) → phase-N.md (текущий этап) → связанные модули → issues/antipatterns. Чем больше проект, тем строже иерархия.

3. **Разделение по агентам.** Контекст архитектора и контекст разработчика — разные. Не смешивать.

4. **Сжатие через спецификацию.** Документация компактнее кода. 100 строк спеки описывают то, что в коде занимает 1000 строк.

5. **Кэширование.** Современные API поддерживают prompt caching — повторяющиеся длинные системные части не пересчитываются. Если у вас длинная неизменная спека и часто меняющиеся задачи — это даёт реальную экономию токенов и времени.

6. **Чистка контекста.** Периодически начинать новый контекст с чистого листа, перенося только результат, а не всю историю отладки.

**Prompt engineering всё ещё нужен** — для отдельных промптов внутри Skills, для формулировки задач агентам. Но без context engineering хороший промпт не спасёт.

## Часть 2. Процесс разработки

### 2.1. Этап 0: Сбор требований

**Рекомендуемый инструмент:** мощная модель уровня Claude Opus или аналог.

**Бюджет:** ~100 рублей на сессию обычно достаточно.

**Примечание по выбору моделей:**

Opus рекомендуется именно для создания первичной документации — здесь важно качество мышления и работа с противоречиями в требованиях. Для дальнейшей работы:

- **Реализация модулей** — Sonnet или аналогичный средний класс
- **Фиксы простых багов** — Sonnet/Haiku или аналоги, чтобы не жечь токены
- **Архитектурные обсуждения** — снова Opus
- **Массовая генерация (тесты, моки, мелочи)** — модели младшего класса

Это не строгое правило, а ориентир. Главное — не использовать Haiku для архитектуры и не использовать Opus для генерации шаблонных тестов. Токены конечны.

**Процесс:**

1. Загружаем ТЗ от заказчика
2. Требуем от модели задать уточняющие вопросы
3. Отвечаем максимально подробно
4. Если нужна серия вопросов — проходим все раунды
5. Получаем общую спецификацию проекта

**Важно:** на этом этапе не торопиться. Качество спецификации определяет всё остальное.

**Роль человека на этом этапе:** не отвечать "как-нибудь", а реально продумывать ответы. Нейросеть не угадает за вас, что значит "удобный интерфейс" или "должно работать быстро". Если на уточняющий вопрос отвечаете "ну, на ваше усмотрение" — получаете на выходе шаблонное решение, не привязанное к вашей задаче.

---

### 2.2. Этап 1: Декомпозиция на этапы

Любой проект, каким бы маленьким он ни был, разбивается на логические этапы.

**Требования к этапу:**

- Можно вручную протестировать результат
- Желательно — заказчик может сам протестировать (обычный человек)
- Этап даёт видимый, ощутимый результат

**Правило первого этапа:**

Первым этапом всегда идёт клиентская сторона. Не база данных, не бэкенд, не инфраструктура — то, что можно пощупать.

Пример для Telegram-бота: первый этап — работающий бот, который отвечает на команды (пусть даже заглушками). Не схема БД, не API-эндпоинты.

Это правило сильно помогает заказчику видеть прогресс и снижает риск "три недели работы, а показать нечего".

---

### 2.3. Этап 2: Test-Driven Development и verification loops

Для каждой функции:

1. Сначала пишем тесты
2. Потом реализуем функцию

**Почему это критично для LLM-разработки:**

Тесты становятся вторым источником истины. У нейросети два ориентира:

- Документация (что делать)
- Тесты (как проверить, что сделано правильно)

Это значительно повышает качество генерации.

---

#### Verification loops: автоматизация цикла

Современная практика идёт дальше классического TDD. Verification loop — это автоматический цикл, в котором агент:

1. Пишет код по спецификации
2. Запускает тесты
3. Читает трейсбэк, если что-то не прошло
4. Фиксит
5. Повторяет, пока не зелёное

В зрелых средах (Claude Code и аналоги) этот цикл работает автоматически. Агент сам видит результат запуска тестов и итерирует. Человек подключается только когда цикл застревает или начинает идти не туда.

**Это работает. Но есть критичные слепые зоны.**

---

#### Слепая зона №1: тесты клиентской стороны

Главная и самая частая проблема в моей практике — LLM почти не покрывает тестами клиентскую сторону.

**Конкретный пример (Telegram-бот):**

Когда пишется бот, нейросеть прекрасно покрывает тестами:

- Backend (FastAPI/aiohttp)
- Базу данных (PostgreSQL, миграции)
- Очереди (Redis, Celery)
- Внешние интеграции

Но почти никогда не покрывает:

- Хэндлеры самого бота
- Логику обработки команд
- Работу с состояниями диалога
- Интеграционные тесты сценариев "пользователь нажал кнопку → получил ответ"

Это огромный пласт логики, который остаётся непротестированным. И именно в нём чаще всего находятся реальные баги, которые видит пользователь.

**Решение:** явно указывать в спеке этапа, какие тесты должны быть. Не полагаться на дефолтное поведение модели. Перед запуском verification loop проверять, что план тестов покрывает в том числе клиентский слой.

---

#### Слепая зона №2: интеграционные тесты ботов

Связано с предыдущим, но шире. Юнит-тесты на отдельные хэндлеры — половина дела. Нужны интеграционные тесты, которые проверяют сквозной сценарий:

- Пользователь пишет /start
- Получает приветствие
- Нажимает кнопку
- Бот делает запрос в БД
- Бот делает запрос во внешний API
- Бот отдаёт результат пользователю

LLM по умолчанию пишет такие тесты крайне редко. Нужно явное требование в спеке.

---

#### Чеклист "слепых зон" верификации

При планировании тестов для этапа явно проверьте:

- [ ] Хэндлеры клиентской стороны (бот, фронт, мобилка)
- [ ] Сквозные интеграционные сценарии "пользователь → ответ системы"
- [ ] Граничные условия (пустые данные, длинные строки, спецсимволы)
- [ ] Состояние при ошибках внешних сервисов (что показываем пользователю)
- [ ] Конкурентные сценарии (одновременные запросы, гонки)
- [ ] Миграции БД (можно ли откатить, не теряя данные)
- [ ] Поведение при перезапуске (восстановление состояния)

Каждый пункт — это то, что LLM регулярно забывает покрывать сама. Человек должен проследить.

---

### 2.4. Этап 3: Итерации и отладка

**При возникновении ошибок:**

1. Документируем ошибку в файле issues.md
2. Решаем ошибку
3. Добавляем решение в тот же файл issues.md
4. Актуализируем главную спецификацию
5. Повторяем, пока все ошибки не решены

**Когда этап завершён:**

Создаём файл result.md:

- Результат тестирования
- Как работает
- Подтверждение, что всё функционирует

Этот файл — маркер завершённого этапа. Его наличие означает: "здесь всё работает, не трогать".

---

### 2.5. Этап 4: Наращивание функционала

Переходим к следующему этапу. Повторяем цикл:

1. Документация этапа
2. Тесты
3. Реализация
4. Отладка (issues → актуализация)
5. Фиксация результата (result.md)

При этом явно указываем: предыдущие этапы не трогать.

---

### 2.6. Управление контекстом при разрастании проекта

Когда проект вырастает до 10+ модулей, невозможно загружать всё в контекст.

**Принцип:** загружаем только то, что связано с текущей задачей.

**Что загружать при работе над новым модулем:**

1. Главную спецификацию (spec.md) — всегда
2. Документацию связанных модулей (по карте модулей)
3. Тесты связанных модулей
4. Issues, если есть релевантные

**Что НЕ загружать:**

- Документацию несвязанных модулей
- Код несвязанных модулей
- Историю решённых issues

**Как определить связанность:**

Смотрим в карту модулей (spec.md). Если модуль A вызывает модуль B или использует его данные — они связаны. Если нет — не загружаем.

Это базовый приём context engineering из раздела 1.7. На малых проектах кажется избыточным, на больших — единственный способ удержать качество.

### 3.2. Формат спецификации (spec.md)

```markdown
# Название проекта

## Обзор
Краткое описание: что делает система, для кого, какую проблему решает.

## Функциональные требования
Нумерованный список (R1, R2, R3...).
Каждое требование — одно предложение.

## Нефункциональные требования
Производительность, безопасность, ограничения.

## Архитектура
Описание компонентов и их взаимодействия.
Схемы — да. Код — нет.

## Карта модулей
Ссылки на документацию всех модулей с кратким описанием.

| Модуль | Документация | Описание |
|--------|--------------|----------|
| Auth | phases/phase-1.md | Авторизация и регистрация |
| API Gateway | phases/phase-2.md | Маршрутизация запросов |
| Payments | phases/phase-3.md | Интеграция с платёжкой |
| External API | docs/api-integration.md | Работа с внешним сервисом |

## Этапы реализации
Список этапов с кратким описанием каждого.

## Версия
Текущая версия и дата последнего обновления.
```

**Зачем нужна карта модулей:**

Когда проект разрастается (10+ этапов), невозможно держать всё в контексте. Карта модулей позволяет быстро найти, какая документация нужна для конкретной задачи. При работе над новой фичей загружаем только связанные модули, а не всё подряд.

В контексте sub-agents карта модулей выполняет ещё одну функцию: она показывает, какие модули можно отдать параллельным агентам без риска пересечения.

### 3.3. Формат документации этапа (phase-N.md)

```markdown
# Этап N: Название

## Цель
Что должно работать по завершении этапа.

## Входные условия
Что должно быть готово до начала этапа.

## Требования
Конкретные требования этого этапа (ссылки на R1, R2... из spec.md).

## Компоненты
Какие части системы затрагиваются.

## Логика работы
Описание словами: что происходит, как данные перетекают,
какие проверки выполняются, какие ошибки возможны.

## План тестирования
Конкретные тесты, которые должны быть. С обязательным указанием
тестов клиентской стороны и интеграционных сценариев.

## Критерии готовности
Как понять, что этап завершён. Что можно протестировать.
```

Раздел "План тестирования" добавлен специально, чтобы закрыть слепые зоны verification loops. Без него LLM по умолчанию покроет backend и оставит фронт/бот без тестов.

### 3.5. Формат antipatterns.md

```markdown
# Antipatterns

Типичные ошибки при генерации/запуске кода.
Каждая ошибка имеет трейсбэк и конкретное решение.

---

## [AP-001] Cannot find module

**Трейсбэк:**
```
Error: Cannot find module 'express'
    at Function.Module._resolveFilename (node:internal/modules/cjs/loader:995:15)
    at Function.Module._load (node:internal/modules/cjs/loader:841:27)
```

**Причина:**
Зависимость не установлена / не указана в package.json.

**Решение:**
Добавить express в dependencies и выполнить npm install.

---

## [AP-002] Cannot read property of undefined

**Трейсбэк:**
```
TypeError: Cannot read property 'id' of undefined
    at getUserData (/app/src/user.js:15:23)
```

**Причина:**
Функция вызывается до получения данных пользователя.

**Решение:**
Добавить проверку на существование объекта перед обращением к свойству.

---

## [AP-003] ...
```

**Что относится к antipatterns:**

- Ошибки импорта/зависимостей
- TypeError, ReferenceError и подобные
- Ошибки подключения к БД
- Проблемы с переменными окружения
- Синтаксические ошибки при генерации
- Любые ошибки с конкретным трейсбэком в логах

**Отличие issues от antipatterns:**

| Issues | Antipatterns |
|--------|--------------|
| Проблемы поведения и логики | Ошибки выполнения кода |
| "Система работает неправильно" | "Система не запускается / падает" |
| Требует анализа архитектуры | Требует исправления кода |
| Может затрагивать спецификацию | Обычно локальное исправление |
| Абстрактное описание | Конкретный трейсбэк |

**Зачем нужен antipatterns.md:**

Нейросеть часто делает одни и те же ошибки. Когда ошибка зафиксирована с трейсбэком и решением, при следующем возникновении:

1. Находим похожий трейсбэк в antipatterns.md
2. Применяем готовое решение
3. Не тратим время на повторный анализ

Со временем antipatterns.md становится базой знаний типичных ошибок проекта. В контексте verification loops это критично: автоматический цикл начинает проходить быстрее, потому что агент сразу видит решение, а не итерирует с нуля.

Отдельным пунктом — тесты клиентской стороны и интеграционные сценарии.

## Часть 4. Skills, MCP и spec-driven development

### 4.1. Spec-Driven Development как методология

То, что описано в этой методичке, в индустрии получило именованную форму — spec-driven development. Подход формализуется в инструментах вроде GitHub Spec Kit, Amazon Kiro и других. Общая идея — единая: спецификация первична, код вторичен.

**Чем AIDD отличается от других реализаций spec-driven development:**

- **Жёсткий запрет кода в документации**, включая псевдокод
- **Разделение issues и antipatterns** как двух разных типов знания
- **DNA Map** для трассировки требование → документация → тест
- **Два режима работы** (Incremental / Rebuild) для разных задач
- **Формула воспроизводимости** как метрика качества методологии
- **Явный фокус на слепых зонах верификации** (клиентская сторона)

Это не значит, что AIDD — конкурент Spec Kit или Kiro. Скорее, это собственная сборка тех же принципов, заточенная под практику фриланс-разработки и работу с разнородными проектами.

---

### 4.2. Что такое Skills

В оригинальной версии методички Skills — это самописные подготовленные промпты. Это рабочее определение, но нужно различать два типа Skills:

**Тип 1: Самописные Skills (slash-команды)**

Промпты, которые вы храните в `.claude/commands/` (или эквиваленте). Используются как `/discover`, `/implement`, `/fix`. Это ваши инструменты под ваши процессы.

**Тип 2: Anthropic Skills**

Формальный механизм, в котором Skill — это пакет: SKILL.md (инструкции) + опциональные исполняемые скрипты + ресурсы. Skill подгружается агентом автоматически, когда задача попадает под его описание.

**Тип 3: Skills из специализированных сервисов**

Существуют сервисы и реестры, где можно найти готовые Skills под конкретные задачи (работа с фреймворками, специфичные интеграции, типовые методологии). Имеет смысл искать готовое перед тем, как писать своё.

**Преимущество всех трёх типов:** промпты в живом общении могут быть неидеальны, человек формулирует как получится. Skills — задокументированы, зафиксированы, отлажены. Они становятся источником истины для повторяющихся действий.

---

### 4.3. Базовые Skills

#### discover.md — Изучение

```markdown
# Discover

## Роль
Ты — аналитик, изучающий проект.

## Задача
Изучить документацию и кодовую базу. Составить понимание:
- Что делает система
- Какие компоненты есть
- Как они взаимодействуют
- Какие зависимости

## Ограничения
- Не вносить изменений
- Не предлагать улучшений (пока не спросят)
- Только анализ и понимание

## Формат ответа
Структурированное описание того, что понял.
Вопросы, если что-то неясно.
```

#### implement.md — Реализация

```markdown
# Implement

## Роль
Ты — разработчик, реализующий функцию по спецификации.

## Задача
Реализовать функцию согласно документации этапа.

## Ограничения
- Строго следовать спецификации
- Не менять существующий работающий код
- Сначала тесты, потом реализация
- Не добавлять функционал сверх описанного
- Тесты обязательно покрывают клиентскую сторону, если она есть

## Входные данные
- Документация этапа
- Существующая кодовая база
- Тесты (если есть)

## Формат работы
1. Подтвердить понимание задачи
2. Написать тесты (включая клиентский слой и интеграционные сценарии)
3. Реализовать функцию
4. Проверить, что тесты проходят
```

#### fix.md — Исправление ошибки

```markdown
# Fix

## Роль
Ты — разработчик, исправляющий ошибку.

## Задача
Найти причину ошибки и исправить её минимальными изменениями.

## Ограничения
- Минимальные изменения (не рефакторить всё вокруг)
- Не менять работающий функционал
- Задокументировать причину и решение
- Перед тем как исправлять — проверить antipatterns.md на похожие случаи

## Формат работы
1. Воспроизвести ошибку
2. Проверить antipatterns.md
3. Найти причину
4. Предложить исправление
5. Внести исправление
6. Проверить, что ошибка исправлена
7. Проверить, что ничего не сломалось
8. Если ошибка типовая — добавить в antipatterns.md
```

#### integrate.md — Интеграция

```markdown
# Integrate

## Роль
Ты — разработчик, интегрирующий новый компонент.

## Задача
Подключить новый компонент к существующей системе.

## Ограничения
- Существующие компоненты не менять (только точки подключения)
- Следовать установленным паттернам проекта
- Минимальные изменения в существующем коде

## Формат работы
1. Изучить интерфейсы существующих компонентов
2. Определить точки интеграции
3. Реализовать подключение
4. Протестировать взаимодействие (включая интеграционные тесты)
```

---

### 4.4. Как использовать Skills

**Для общения и обсуждения:** свободная форма, диалог с LLM.

**Для конкретных действий:** использовать Skills.

Пример:

- Обсуждаем архитектуру — свободный диалог
- Реализуем функцию — вызываем `/implement` с документацией этапа
- Нашли баг — вызываем `/fix` с описанием ошибки

В среде с поддержкой sub-agents Skills работают в комбинации: агент Architect использует свои Skills, агент Developer — свои. Конфигурация хранится в репозитории, переезжает с проектом, передаётся новым участникам.

---

### 4.5. MCP (Model Context Protocol)

MCP — это протокол, через который LLM-агент получает доступ к внешним инструментам и данным: файловой системе, API сервисов, базам данных, дизайн-инструментам и так далее. С точки зрения этой методички MCP — инфраструктура для context engineering: способ доставлять в контекст не только текст из документации, но и данные из внешнего мира.

**Зачем это нужно AIDD:**

- Агент может читать состояние внешних сервисов без копирования данных вручную
- Агент может работать с дизайнерскими макетами как с источником спецификации
- Документация может ссылаться на живые источники, а не статические снимки

**Из практики (Figma MCP):**

Я использовал MCP-интеграцию с Figma для генерации мобильного приложения на Flutter из макета. Это серьёзно экономит время — не приходится верстать руками экран за экраном, LLM сама считывает структуру макета и генерирует UI.

Но есть конкретные ограничения, которые видны только в практике:

- Если на странице макета есть основной экран и его вариации (модальные окна, всплывающие состояния, альтернативные виды) — LLM берёт **только основной экран**. Вариации, модалки и второстепенные состояния она додумывает сама, а не считывает из макета.
- Если есть список разделов или табов — LLM часто реализует только первый, остальные пропускает или делает заглушками.
- Многократно приходится самому проверять и явно указывать "вот этот экран ты не сделал, вот этот пропустил".

**Вывод:** MCP не заменяет ручную проверку. Это инструмент, который ускоряет, но не закрывает полностью задачу. Человек всё ещё должен проверять, что в результат попало всё, что должно было попасть.

**Какие MCP-серверы имеет смысл подключать в AIDD-проекте:**

- Доступ к дизайн-системам (Figma и аналоги) — для проектов с UI
- Доступ к таск-трекерам — если они источник требований
- Доступ к документации внешних API — чтобы не копировать вручную
- Доступ к мониторингу/логам — для дебага в продакшене

Принцип тот же, что в context engineering: подключаем только то, что реально нужно для текущих задач, а не всё подряд.

---

## Часть 5. Git и работа в команде

### 5.1. Документация как часть репозитория

Документация — это такая же часть проекта, как и код. Она живёт в репозитории, версионируется, ревьюится.

**Правило:** всё, что влияет на проект, должно быть в Git.

Это включает не только spec.md и phase-N.md, но и конфигурацию sub-agents (`.claude/agents/`), Skills, MCP-конфигурацию. Всё, что определяет, как LLM работает с проектом, должно быть воспроизводимо.

---

### 5.2. Ветвление

**Основной принцип:** один этап = одна ветка.

```
main
├── feature/phase-1-auth
├── feature/phase-2-api
├── feature/phase-3-payments
└── ...
```

**При командной работе:**

- Каждый разработчик работает в своей ветке
- Если фичи независимы (разные модули/сервисы) — параллельные ветки от main
- Если фичи зависимы — последовательно, от предыдущей ветки
- Merge в main только после прохождения тестов и ревью документации

**При работе с параллельными sub-agents:**

Тот же принцип, но более жёстко. Каждый sub-agent работает в своей ветке, чтобы их изменения физически не пересекались до момента ревью. Один агент = один модуль = одна ветка. Это упрощает откаты и конфликт-резолюшн.

---

### 5.3. Коммиты

**Формат:**

```
[phase-N] Краткое описание

- Что сделано
- Какие файлы документации затронуты
```

**Примеры:**

```
[phase-2] Реализован API Gateway

- Добавлена маршрутизация запросов
- Обновлена spec.md (карта модулей)
- Создан phases/phase-2.md
```

```
[fix] Исправлена авторизация при истёкшем токене

- Добавлен кейс в issues.md
- Актуализирован phase-1.md
```

---

### 5.4. Командная работа

Методичка работает и для команды. Документация — общий источник истины.

**Как это работает:**

1. Один человек сделал этап → задокументировал → закоммитил
2. Другой человек видит всю документацию в репозитории
3. Он дополняет документацию по своей части
4. Всё фиксируется в Git, всё отслеживается

**Разделение работы:**

- Архитектор (или senior) — пишет spec.md и определяет этапы
- Разработчики — реализуют этапы, пишут phase-N.md
- Все — дополняют issues.md при обнаружении проблем
- Все — актуализируют документацию при изменениях

**Git спасает:**

Если нейросеть что-то сломала — откатываемся. Поэтому частые коммиты обязательны. Лучше много маленьких коммитов, чем один большой. Особенно при работе с параллельными агентами: чем мельче атомарные коммиты, тем проще понять, кто из них что сломал.

## Резюме

**Роль человека** — понимать архитектуру, управлять системой. Кодинг знать не обязательно. AIDD не работает без квалифицированного человека-оркестратора — это структурное свойство методологии, а не временное ограничение моделей.

**Контекст — главное оружие.** Экономим его: пишем суть, не пишем код и псевдокод. Управление контекстом (context engineering) — отдельная дисциплина, которая важнее prompt engineering.

**Документация — источник истины.** Одна версия, всегда актуальная, живёт в Git.

**Тесты — второй источник истины.** TDD обязателен. Verification loops автоматизируют цикл, но не закрывают слепые зоны — человек должен явно требовать тесты клиентской стороны и интеграционные сценарии.

**Разделение ролей** — один контекст = одна ответственность. Sub-agents формализуют это на уровне инфраструктуры, но границы между ними определяет человек.

**Модульность — ключ к контролю.** Маленькие сервисы легче генерировать и держать в контексте. И только модульный проект эффективно параллелится между sub-agents.

**Skills — отлаженные инструкции.** Самописные slash-команды, Anthropic Skills, готовые из реестров — все три типа имеют место. Для действий используем их, не импровизируем.

**MCP — инфраструктура context engineering.** Доставляет внешние данные в контекст, но не заменяет ручную проверку. LLM через MCP всё равно пропускает вариации и второстепенные элементы.

**Spec-Driven Development — индустриальное название** того, что мы делаем. AIDD — собственная сборка этого подхода с фокусом на запрет кода в спеке, разделение issues/antipatterns и слепые зоны верификации.

**Не трогать работающее.** Явно указываем нейросети границы изменений.

**Git спасает.** Частые коммиты, один этап = одна ветка, один sub-agent = одна ветка.

## Дополнение: Definitions (Определения)

### Что считается кодом в документации

Под кодом понимается любая конструкция, которую LLM может воспринять как директивную реализацию:

- Код на любом языке
- Псевдокод
- SQL-запросы
- Регулярные выражения
- Конфиги, похожие на готовые файлы (Dockerfile, compose, CI)
- Алгоритмические конструкции (if/for/return и аналогичные)

Это уточнение усиливает правило "без кода", но не отменяет его.

### Итерация

Итерацией считается полный цикл:

1. Генерация или изменение кода
2. Запуск в чистом окружении
3. Обнаружение ошибки
4. Фиксация в issues.md или antipatterns.md
5. Актуализация документации
6. Повтор

### Verification loop

Verification loop — это автоматический цикл "пишу код → запускаю тесты → читаю трейсбэк → фикшу → повторяю", который отрабатывает агент без участия человека. Является частной формой итерации, в которой человек подключается только при застревании цикла.

### Воспроизводимость

Воспроизводимость — это способность получить работающий продукт в чистом окружении исключительно из документации и тестов без участия автора оригинального кода.

### Sub-agent

Sub-agent — отдельный LLM-агент с изолированным контекстом, запускаемый в рамках общей рабочей сессии. Имеет свою роль (архитектор, разработчик, фиксер) и работает в своих границах модулей. Конфигурация sub-agents хранится в репозитории.

### Context engineering

Context engineering — управление содержимым контекстного окна LLM на протяжении работы над проектом. Включает решения о том, что загружать, что выгружать, как разделять информацию между агентами, когда чистить контекст, как использовать кэширование. Отличается от prompt engineering, который касается формулировки отдельных запросов.

## Дополнение: Два режима разработки

Чтобы методология не противоречила сама себе, вводятся два режима работы:

### Incremental Mode

- Добавление новой функциональности
- Минимальные изменения
- Строгое правило "не трогать работающее"
- Локальные изменения

### Rebuild Mode

- Полная пересборка проекта с нуля
- Допускается другая реализация
- Истина — документация и тесты
- Проверяется воспроизводимость

## Дополнение: DNA Map (docs/dna-map.md)

DNA Map связывает требования, документацию и тесты.

Для каждого требования должно быть понятно:

- Где оно описано
- Где реализовано
- Каким тестом проверяется

Если требование нельзя связать с тестом — оно считается незащищённым. Если незащищённое требование относится к клиентской стороне — это красный флаг (см. слепые зоны verification loops).

## Дополнение: Benchmark (docs/benchmark.md)

### Цель

Зафиксировать протокол воспроизводимости проекта.

### Quality Gates

Проект считается воспроизведённым, если:

- Контейнеры поднимаются
- Нет runtime-ошибок
- Все тесты проходят (включая тесты клиентской стороны)
- Нет ручных действий вне описанных процедур

### Метрика итераций

L1 (простые проекты):
- 0 — отлично
- 1–2 — хорошо
- 3+ — требуется доработка документации

L2 (интеграции):
- 0–1 — отлично
- 2–3 — хорошо
- 4+ — требуется доработка

## Дополнение: Формула воспроизводимости

Воспроизводимость = Полнота документации × Покрытие тестами × Качество декомпозиции × Дисциплина человека

Если любой множитель близок к нулю — воспроизводимость стремится к нулю.

Дисциплина человека добавлена как четвёртый множитель в v2 не для красоты. Без человека, который соблюдает методологию (не лезет в работающий код, не позволяет sub-agents пересекаться, требует тесты клиентской стороны, актуализирует документацию) — все остальные множители обнуляются на длинной дистанции.

Это и есть главный смысл AIDD: методология даёт структуру, инструменты дают скорость, но финальное качество всегда упирается в человека-оркестратора.
